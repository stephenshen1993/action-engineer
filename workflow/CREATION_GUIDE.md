# Action Engineer 创作指南

## 作用

这是日常创作时提供给 ChatGPT 的统一规则文件。

目标：稳定发布真实经历型抖音长文，降低创作阻力，在文章达到可发布状态后及时停止修改。

## 协作分工

| 角色 | 职责 |
| --- | --- |
| 用户 | 提供真实经历、判断、情绪和最终决策 |
| ChatGPT | 深聊、挖掘核心矛盾、输出可发布文章 |
| Codex | 归档文章、素材和索引，维护项目状态 |

## 创作输入

开始一篇新文章时，优先提供：

```text
真实现场：
当时最真实的感受：
核心矛盾：
最想表达的一句话：
必须保留的细节：
需要规避的隐私风险：
```

不要长时间讨论选题和结构。优先在 30 分钟内形成一个可发布初稿。

## 写作规则

### 内容气质

- 真实
- 克制
- 有现场感
- 有普通人的犹豫和不确定
- 不鸡汤
- 不营销
- 不贩卖焦虑
- 不把 AI 神化
- 不塑造导师人设

更像朋友之间的真实聊天，而不是一堂总结课。

### 写作顺序

1. 先找到已经发生过的真实现场和真实情绪。
2. 优先保留具体事件，再从经历中提炼核心矛盾。
3. 技术细节只服务故事和情绪。
4. 先真实，再深刻。

不要先预设大道理，再寻找故事证明。

### 已验证经验

不要从观点出发。

优先：

- `因为真的有点顶不住了`
- `咱们这又一个小时过去了`
- `查了一下午`
- `算时间`
- `又一个小时过去了`

避免先写：

- `我进入第二关了`
- `我开始长期主义了`
- `我担心的东西变了`

过早追求深刻，容易脱离真实经历。文章最有价值的部分通常来自真实困惑，而不是总结结论。

不要过早升华。慎用：

- `我发现`
- `我意识到`
- `原来`
- `坚持`
- `成长`
- `长期主义`

### 常见问题

避免：

- 总结腔、过度工整排比、强行上价值
- 只交代事件，没有真实情绪
- 强行燃、强行逆袭、强行制造焦虑
- 堆专业名词，写成技术方案复盘

判断标准：

如果读起来像一堂课，而不是一个人真的经历过，就要改。

### 标题

标题更偏：

- 生活化
- 观察感
- 场景感
- 自我发现

避免：

- 口号
- 夸张焦虑
- 强行制造冲突
- 连续重复同一种句式

## 停止重写

出现以下任一信号时，停止重写，进入 Review：

- 开始像作者本人说的话
- 开始出现共鸣
- 开始纠结细节
- 围绕同一个问题讨论超过 30 分钟

不要继续推进 V9、V10、V11。

## 单次 Review

正文只执行一次集中 Review。只检查：

1. 是否存在事实错误？
2. 是否存在隐私、客户或同事关系风险？
3. 是否存在普通读者无法理解的明显歧义？
4. 正文是否偏离本篇核心矛盾？

输出格式：

```text
结论：发布 / 修改一次后发布

阻塞项：
- 没有则写“无”
- 有则逐条指出，并给出最小修改建议
```

Review 后只集中修改一次。

以下问题不阻塞当天发布：

- 还能更顺
- 还能更有情绪
- 可能略有 AI 味
- 或许可以换一种结构
- 标题也许存在更优解

## 时间盒

单篇文章总耗时上限为 90 分钟：

```text
5 分钟：整理真实现场
30 分钟：ChatGPT 深聊并形成可发布初稿
15 分钟：执行一次阻塞项 Review
20 分钟：集中修改一次
5 分钟：从最多 3 个标题中选择 1 个
10 分钟：交给 Codex 归档
5 分钟：预留缓冲
```

## 发布后归档

Codex 负责：

1. 在 `content/published/` 新增文章归档。
2. 将实际使用的图片放入 `content/assets/published/<文章文件名>/`。
3. 更新 `content/published/INDEX.md`。
4. 按需更新 `CURRENT_STATUS.md`。
5. 将关键复盘追加到 `logs/project_log.md`。

归档模板：

```text
workflow/templates/published_article.md
```

视觉规则：

```text
workflow/VISUAL_GUIDE.md
```

## 当前不要做

- 自动化编排
- 多 Agent
- RAG 或向量数据库
- 数据看板
- 多平台内容矩阵
- 复杂版本管理

当前真正主线只有一句话：

> 持续记录一个真实工程师的变化过程。
