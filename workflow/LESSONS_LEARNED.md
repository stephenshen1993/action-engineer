# 创作复盘

## 2026-06-02 第17篇《本来想解决一个问题，结果事情越来越不对劲》

### 文章质量

文章质量提升的关键通常不是增加观点，而是增加：

- 故事
- 情绪
- 真实过程

减少：

- 总结
- 方法论
- 说教

优先记录：

- 事情开始不对劲的时刻
- 困惑出现的时刻
- 怀疑出现的时刻
- 荒诞感出现的时刻

这些通常比最终结论更有价值。

工程师共鸣来源：

> 卧槽，这不就是我吗

而不是：

> 这个人懂得真多

当文章开始变成长篇认知分析时，优先追问：

> 当时发生了什么

而不是：

> 最后得出了什么结论

### 标题选择

标题优先级：

> 故事感 > 悬念感 > 观点表达

不要按散文标准选标题。优先按抖音点击逻辑选标题。

### 停止重写

出现以下情况立即进入发布流程：

- 核心矛盾明确
- 情绪真实
- 开始纠结字句
- 开始讨论 V3、V4、V5

### 长期协作风险

ChatGPT 容易出现“基线版本锁死”：

- 死抱旧标题
- 死抱旧结构
- 死抱旧结论

后续协作中发现此问题时，直接放弃旧基线，以最新结论为准。

## 2026-06-04 第18篇《我明明想找个助手，最后却当了两天监工》

### 选题获取

本次没有从抽象观点出发。

选题直接来自最近 48 小时真实经历：

- Codex 重装
- 自动审查模式
- Claude Code 落地
- 网络质量评估需求 Review

验证：

> 真实现场比刻意寻找选题更容易形成内容。

### 工具题材风险

创作过程中多次差点滑向：

> Claude Code 好用，所以写工具测评。

最终确认：

工具只是故事。

变化才是主题。

当选题涉及 AI 工具时，不要写工具本身，优先写工具带来的变化。

### 内容阶段变化

第 18 篇标志着项目进入新的内容阶段。

前期内容：

> AI 改变我的思维。

当前内容：

> AI 开始参与我的工作。

后续可扩展方向：

- AI 同事感
- AI 信任建立
- AI 协作模式
- Agent 时代的工程师
- 控制欲与放权
- AI Review 体验
- 工程师如何与 Agent 协作

### 视觉经验

再次确认：

- 封面负责点击
- 头图负责现场感

不要混用。

本次最终头图方向：

> 程序员后仰坐在工位前，AI 持续执行任务，人物处于观察状态。

表达：

> 终于不用每一步都自己干了。

### 原生表达

本次最有价值的真实表达：

> 原来不是它不会干。
>
> 是我一直没让它干。

真实吐槽往往比总结性观点更有力量。

创作时优先保留真实表达。
