# Action Engineer 视觉指南

## 核心原则

视觉服务于内容。

不追求炫酷。

不追求AI感。

不追求科技感。

优先保证：

- 真实感
- 一致性
- 长期可持续

账号视觉统一的核心不是固定工位。

而是：

> 一个工程师在思考。

不要为了统一而统一。

统一的是人格气质，不是固定画面模板。

标题、封面、头图不要重复表达同一个信息。

三者分工：

- 标题负责制造停顿
- 封面负责提供点击理由
- 头图负责提供现场感和情绪

---

# 封面

职责：

让用户停下来。

封面负责点击，不负责完整表达现场。

封面可以放大矛盾、强化冲突，但不要承担正文解释任务。

## 默认模板（80%文章）

元素：

- 真实工位
- 暖色灯光
- 夜晚环境
- 电脑/笔记本
- 生活痕迹

要求：

- 文字 2~4 行
- 大字
- 少元素
- 可以直接使用反直觉标题
- 优先具体冲突，不要过度文学化或抽象化
- 不出现机器人
- 不出现未来城市
- 不出现赛博朋克
- 避免科技海报感

封面文案优先使用具体动作和具体冲突。

优先：

- `明天上线，我却重做了`
- `做了两周，我又推翻了`
- `代码快写完了，我却慌了`

慎用：

- `差一点`
- `做不下去了`
- `说服不了`

抽象表达可以有气质，但通常不如具体冲突更容易让用户停下来。

## 封面类型

### A 类：工位现场

适用：

- 技术问题
- 排查过程
- 项目事故
- 代码、上线、Review 等明确工作现场

### B 类：情绪现场

适用：

- 认知变化
- 情绪发现
- 自我观察
- 文章核心不是“在哪里发生”，而是“人当时是什么状态”

可以使用：

- 黄昏
- 城市
- 阳台
- 发呆
- 留白

这类画面可以脱离工位，只要能承接文章情绪。

### C 类：隐喻现场

适用：

- 世界观变化
- 阶段转折
- 重要节点
- 需要比真实场景更强的象征感

## 封面候选方案：大概念字

适用：

- 偏认知
- 偏荒诞
- 偏思考型内容
- 文章核心概念很强

可以尝试：

- 大概念字
- 杂志封面风
- 内容品牌风
- 人物缩小
- 增加留白

例如把 `需求` 作为主视觉概念，而不是只做“人物 + 大字”的传统自媒体封面。

这种方案不作为默认模板，但可以作为偏认知内容的候选方向。

## 封面设计感补充：不要停在工位配图

当文章是偏认知、偏荒诞、偏思考型内容时，封面不要只做：

> 工位照片 + 大字

这类封面虽然安全，但容易缺少设计感和视觉记忆点。

应优先寻找文章的视觉锚点，例如：

- `需求`
- `0 行`
- `提测`
- `问题`
- `延期`

把这个视觉锚点做成封面主概念，而不是让人物工位承担全部表达。

判断封面是否足够好时，除了检查真实感和文字清晰度，还要检查：

1. 是否有一个明确视觉概念？
2. 是否在缩略图里能一眼记住？
3. 是否只是普通工作照？
4. 文字是否只是贴在图上，而不是参与设计？
5. 是否和近期高质量封面的设计感保持一致？

对于强概念文章，可以优先尝试：

- 明亮或半明亮色系
- 杂志封面排版
- 大概念字
- 主体缩小
- 大面积留白
- 单一视觉锚点

不要默认所有程序员文章都做成深夜暗黑工位风。

# 封面（重大节点）

适用：

- 世界观升级
- 人生转折
- 重大认知变化

允许：

- 人物
- 剪影
- 电影感

禁止：

- AI导师感
- 成功学海报感

# 头图

职责：

进入文章后的情绪铺垫。

头图负责现场感，不负责点击。

原则：

- 不重复标题
- 不解释内容
- 只提供情绪
- 服务文章核心现场，而不是服务主题关键词

优先场景：

- 工位
- 窗边
- 夜景
- 路灯
- 咖啡馆
- 地铁

允许无文字。

选择头图时，先判断文章真正发生在哪里。

如果核心现场是朋友聊天，就优先表现“一个人在认真回复消息”，而不是强行表现 AI、代码、程序员工位或科技感元素。

头图不是关键词配图。

它应该让读者进入文章发生的那个时刻。

头图继续坚持：

- 记录现场
- 记录瞬间
- 记录情绪

不要试图解释文章。

如果文章核心是“第三遍看文档时开始觉得不对劲”，头图就应该表现那个具体瞬间，而不是解释指标、链路或需求结构。

头图也可以服务情绪现场，而不只是事故现场。

如果文章真正的情绪核心是疲惫、荒诞、发呆、破防或自我观察，头图可以完全脱离工位。

优先表达文章真正的情绪核心，而不是重复文章发生地点。

当文章讨论 AI Agent 协作时，可以优先表现：

- 程序员处于观察、后仰、Review 或等待状态
- AI 或屏幕任务在持续执行
- 人物不再每一步都亲自操作
- 人在看活，而不是亲自干活

重点表达工作关系的变化，而不是展示工具界面。

# 插图

默认不使用。

仅在确有必要时使用。

每篇最多2张。

职责：

补充情绪或场景。

不是解释内容。

# 配乐

抖音长文配乐不等于短视频 BGM。

文章是主角。

音乐是配角。

配乐职责是降低纯文字阅读的枯燥感，而不是推动情绪。

高优先级原则：

- 存在感越低越好
- 避免明显高潮
- 避免强节奏推进
- 宁可平淡，不要抢戏

优先：

- 轻氛围
- 弱节奏
- 低存在感
- 无明显高潮
- 环境感

谨慎使用：

- 史诗感
- 强情绪
- 强推进
- 强旋律记忆点

创作口诀：

> 封面负责点击。
>
> 标题负责停留。
>
> 开头负责留下。
>
> 正文负责共鸣。
>
> 配乐负责别添乱。

选音乐前先判断文章整体停留在哪种状态。

## A 类：问题获得解决

适用：

- 想明白了
- 获得答案
- 完成成长
- 问题已经被解决

可以考虑：

- `Nuvole Bianche`
- `Una Mattina`
- 其他低存在感、有收束感的音乐

## B 类：问题仍然悬着

适用：

- 刚发现问题
- 开始怀疑
- 开始慌张
- 问题尚未解决
- 结尾停在不确定里

这类文章应选择更克制、更悬而未决、更偏思考感的音乐。

避免使用明显带有“解决问题”“获得答案”气质的配乐。

配乐服务全文阅读体验，不只服务最后一句情绪。

如果最后一句很气，但全文停留在“发现问题”而不是“发泄问题”，音乐仍应保持克制和思考感。

配乐也可以服务文章的认知变化曲线。

如果文章情绪是“怀疑 → 验证 → 确认”，可以选择有推进感但不过度胜利的音乐。

`Experience` 更适合这类认知推进、验证和确认的过程，不默认代表问题已经解决。

不要只追求好听。
