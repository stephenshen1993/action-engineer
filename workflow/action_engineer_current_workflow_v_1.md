# Action Engineer 当前工作流（V1）

> 当前阶段目标：
>
> 不追求“最先进 AI 工作流”。
>
> 而是：
>
> 建立一种可长期持续的人机协作方式。

---

# 当前核心原则

当前最大的风险不是：

- AI 不够强
- 工具不够先进
- 工作流不够复杂

而是：

> 过早系统化。

所以当前阶段：

- 不追求复杂 Agent
- 不追求自动化编排
- 不追求 AI 工厂流水线
- 不追求完美系统

当前真正重要的是：

> 持续输出真实内容。

---

# 当前整体工作流

```text
填写单篇创作卡
    ↓
ChatGPT 深聊与认知推演
    ↓
形成可发布初稿
    ↓
执行一次阻塞项 Review
    ↓
集中修改一次
    ↓
发布内容
    ↓
Codex 沉淀项目资产
```

---

# 7 天轻量工作流试运行

当前优先解决的问题不是缺少选题，而是文章已经可以发布后仍持续修改。

单篇文章总耗时上限：

```text
90 分钟
```

时间分配：

```text
5 分钟：填写单篇创作卡
30 分钟：ChatGPT 深聊并形成可发布初稿
15 分钟：执行一次阻塞项 Review
20 分钟：集中修改一次
5 分钟：从最多 3 个标题中选择 1 个
10 分钟：交给 Codex 归档
5 分钟：预留缓冲
```

初稿后只允许：

1. 一次集中 Review。
2. 一次集中修改。
3. 从最多 3 个标题中选择 1 个。

Review 只拦截：

1. 事实错误。
2. 隐私、客户或同事关系风险。
3. 普通读者无法理解的明显歧义。
4. 正文偏离本篇核心矛盾。

“还能更顺”“还能更有情绪”“可能略有 AI 味”“标题也许还有更优解”不阻塞当天发布。

---

# 各工具职责

| 工具 | 当前职责 |
|---|---|
| ChatGPT | 深度思考 / 内容推演 / 最终文章输出 |
| Codex | 项目维护 / 文件沉淀 / 长期上下文 |
| Git | 记录认知演化 |
| Markdown | 长期资产载体 |
| 抖音 | 获取真实反馈 |

# 阶段1：现实世界中的灵感

灵感来源：

- 工作中的某个瞬间
- AI 使用体验
- 焦虑
- 深夜思考
- 内容创作卡顿
- 对未来的担忧
- 工程师惯性

此阶段重点：

> 不分析。
>
> 先记录。

记录方式：

```text
content/ideas/
```

文件示例：

```text
2026-05-25-正确性陷阱.md
```

只记录：

- 一个感觉
- 一个矛盾
- 一个观察
- 一句话

不要追求完整。

---

# 阶段1.5：填写单篇创作卡

正式开始创作前，使用：

```text
workflow/templates/article_creation_card.md
```

只记录：

- 真实现场
- 核心矛盾
- 最想表达的一句话
- 必须保留的细节
- 需要规避的隐私风险

不要在选题和结构讨论中停留太久。

---

# 阶段2：ChatGPT 深聊（核心阶段）

这是当前最重要的阶段。

因为当前项目真正有价值的东西是：

> 真实思考过程。

在 ChatGPT 中：

- 深聊
- 推演
- 怀疑
- 自我观察
- 校准表达
- 挖真实情绪
- 挖核心矛盾
- 提炼长期主题

此阶段更像：

```text
认知实验室
```

注意：

不要急着让 Codex 接手。

因为这个阶段的内容通常：

- 很模糊
- 很发散
- 还未结构化

---

# 阶段3：提炼“结构化认知”

当深聊逐渐清晰后：

会开始出现可沉淀内容。

例如：

## 1. 一个明确主题

```text
正确性陷阱
```

## 2. 一个核心矛盾

```text
很多工程师的问题不是不努力，
而是总想先找到更正确的开始方式。
```

## 3. 稳定认知

例如：

- 过早系统化
- workflow 成瘾
- 长期准备
- 行动阻力
- 工程师惯性

## 4. 一个明确内容方向

例如：

- 适合写成长文
- 可以延展成系列
- 可以沉淀到 ideas

这时候：

才进入 Codex 协作阶段。

---

# 阶段4：ChatGPT 输出最终文章

当前阶段：

> ChatGPT 负责最终文章输出。

原因：

目前 ChatGPT 更擅长：

- 活人感
- 节奏感
- 情绪连续性
- 留白
- 长文推演
- 微妙表达
- 真实感

尤其适合：

```text
真实工程师长期记录型内容
```

当前内容重点：

- 不鸡汤
- 不营销
- 不像 AI 创业导师
- 不强行升华
- 更像真实人在说话

初稿形成后，使用：

```text
workflow/article_review_prompt.md
```

只执行一次阻塞项 Review，再集中修改一次。

---

# 阶段5：Codex 沉淀项目资产

文章或认知成熟后：

交给 Codex 做项目级维护。

Codex 当前主要负责：

- 创建 md 文件
- 更新 TODO
- 更新 CURRENT_STATUS
- 维护 `content/published/INDEX.md`
- 更新 project_log
- 管理项目结构
- 维护长期上下文

核心目标：

> 把“聊天”变成“长期资产”。

这是当前 Codex 最重要的价值。

---

# 阶段6：发布内容

当前平台重点：

- 抖音长文

当前阶段目标：

> 建立持续输出惯性。

而不是：

- 爆款
- 数据焦虑
- 商业化
- 内容矩阵
- 工业化生产

---

# 阶段7：简单复盘

复盘位置：

```text
logs/project_log.md
```

每天只需要几句话。

例如：

```md
# 2026-05-25

今天完成：

- 初始化 action-engineer
- 开始 Codex 协作
- 建立 AI-native 工作流

今天最大的感受：

真正重要的，
还是持续输出。
```

不要过度数据化。

7 天试运行期间只额外记录：

- 总耗时是否超过 90 分钟
- 初稿后是否修改超过 1 次
- 是否完成发布

将结果追加到：

```text
logs/project_log.md
```

当前阶段不需要：

- KPI系统
- 自动分析
- 数据看板
- 复杂标签体系

---

# 当前明确不要做的事情

至少未来 1~2 个月：

尽量不要深陷：

- MCP
- 多 Agent
- 自动化编排
- RAG
- 向量数据库
- AI 中台
- 内容流水线
- 超复杂 workflow

因为这些东西：

非常容易重新触发：

> “先把系统设计正确”

的工程师惯性。

---

# 当前真正主线

当前真正的主线只有一句话：

> 持续记录一个真实工程师的变化过程。
